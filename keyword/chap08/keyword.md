- OAuth2 방식
    - OAuth2란 무엇일까요?
        - 사용자의 비밀번호를 제3자의 애플리케이션에 공유하지 않고, 특정 서비스에 대한 접근 권한을 위임하는 것이다.
    - Token이란 무엇일까요?
        - 토큰은 권한을 증명하기 위해 사용되는 암호화된 문자열이다.
    - OAuth2에서 사용되는 토큰은 어떤 것이 있고, 각각 어떤 용도일까요?
        1. Access Token
            - 리소스에 직접 접근할 때 사용하는 토큰이다.
            - API 호출시에 HTTP헤더에 해당 엑세스 토큰을 포함하여 보낸다.
            - 보안을 위해 생명주기가 매우 짧은 편이다.
        2. Refresh Token
            - 만료된 엑세스 토큰을 새로 발급받기 위해 사용하는 키이다.
            - 생명 주기가 길다.
            - 리프레쉬 토큰이 있기 때문에 엑세스 토큰이 만료될 때마다 다시 로그인하는 번거로움은 없다.
    - OAuth2 방식의 장점과 단점은 무엇일까요?
        - 장점
            - 실제 비밀번호가 노출되지 않기 때문에 보안이 강력하다.
            - 세분화된 권한 범위를 지정할 수 있다.
            - 리프레쉬 토큰이 있어 매번 로그인하는 번거로움이 없다.
            - 구글, 네이버, 카카오 등이 해당 표준을 지원하여 상호 호환성이 높고 표준화 됨
        - 단점
            - 단순 ID/PW보다 복잡한 구현 구조를 가진다.
            - 토큰 탈취의 보안 문제가 존재한다. (엑세스 토큰을 가로채 만료 전까지 사용 가능)
            - 리프레쉬 토큰 보안에 큰 신경을 써야한다.
- Cookie와 Session
    - Cookie란 무엇일까요?
        - 쿠키란 클라이언트 로컬에 저장되는 키와 값이 들어있는 작은 데이터 파일이다.
        - 서버가 사용자의 브라우저에 정보를 저장해두면, 사용자가 해당 웹사이트를 방문할 때마다 브라우저가 해당 정보를 서버에 함께 전송한다.
        - 속도가 세션보다 빠르다.
        - 구성 요소 : 이름, 값, 유효시간, 도메인, 경로
        - ex) 아이디 자동 완성, 장바구니, “오늘 이 창을 다시 보지 않음” 등
    - Session이란 무엇일까요?
        - 쿠키를 기반으로 하지만, 사용자 정보를 브라우저가 아닌 서버 측에서 관리한다.
        - 사용자에게는 그 정보에 접근할 수 있는 “세션 ID” 정보만 제공한다.
        - 보안 면에서 쿠키보다 우수하지만, 사용자가 많아질수록 서버 메모리를 많이 차지하게 된다.
        - 브라우저 종료 시 삭제된다.
        - ex) 로그인 상태 유지
- JWT
    - JWT (JSON Web Token)은 무엇일까요?
        - JSON 객체에 인증에 필요한 유저ID, 권한, 만료 시간 등을 모두 담은 후 비밀 키를 사용하여 서명한 것이다.
        - API를 호출할 때마다 비밀 키를 사용해 암호화한 JWT 토큰을 헤더에 담아 클라이언트에 보낸다.
        - 로컬에 저장하기 때문에 서버 용량에 상관 없고 보안에 유리하다
        - 모바일 앱에서 사용하기 적합하다
            - 많은 플랫폼 및 기기에서 동작하는데 JWT를 사용하면 플랫폼 독립적으로 사용자 인증을 처리할 수 있다.
        - 토큰 만료 처리를 구현해야 하는 단점이 있다.
    - JWT 방식의 토큰은 어떤 구조로 되어있고, 각각 어떤 용도로 사용되나요?
        1. Header:
            - 토큰의 데이터를 담는 역할을 한다
            - 토큰의 타입을 담는 typ, 서명 알고리즘인 alg가 있다.
        2. Payload:
            - 실제로 전달하려는 데이터를 담는 용도이다.
            - 등록된 클레임: iss(발급자), exp(만료 시간), sub(제목/유저ID)
            - 비공개 클레임: 개발자가 임의로 넣고 싶은 추가 정보(UserId, email) 등으로 구성
        3. Signature:
            - 이 토큰이 변조되지 않았음을 보증하는 도장역할으로
            - 토큰의 무결성을 검증한다.
            - 인코딩된 Header + 인코딩된 Payload + 서버만 아는 비밀 키(Secret Key)를 합친 뒤, Header에 선언됐던 알고리즘(alg)로 암호화하여 생성한다.
